# i18n 架构红线 vs 翻译完整度解耦——v0.1 ship stub 模式

> 原决策 20（DESIGN.md v0.6 §13 附录）

i18n 是典型"架构早立、翻译可增量"的工程问题。决策把这两件事分开：v0.1 强制所有用户可见字符串走 `t()` lookup（红线），但 `zh.json` 只覆盖关键 key（导航、核心动作按钮、错误信息、欢迎屏文案），其余 fallback 到 `en.json`。v0.1.5 补齐 `zh.json` 至 100% 对齐，CI 强制校验。这避免了"为了上中英两套翻译延后 MVP 发布"，又保住了"事后改造 i18n 架构"的成本红线。
